## Codebase Patterns
- Shell scripts must use LF line endings (enforced by .gitattributes `*.sh text eol=lf`)
- Use `jq` to safely modify manifest.json in scripts
- manifest.json uses `version` for Chrome's strict X.Y.Z format and `version_name` for custom display format
- The Write tool in Claude Code may produce CRLF; always verify/fix with `sed -i 's/\r$//'` or rely on .gitattributes
- Project root: manifest.json, background.js, modules/, shared/, vendor/, config/, scripts/
- Use `chrome.runtime.getManifest()` to read manifest fields from extension pages (popup, options, background)
- No build toolchain or linter — quality checks are JSON validation and manual review
- bump-version.sh uses sed for in-place manifest edits (preserves formatting) and jq for validation only
- VERSION file at project root contains the custom version string (X.XX.XXXXy format)
- GitHub Actions workflows live in `.github/workflows/`; use `::error::` prefix for inline PR annotations
- Branch protection rules (blocking PR merges on failed checks) are configured in GitHub repo settings, not workflow files
- Use `[skip ci]` in commit messages from automated workflows to prevent infinite CI loops
- Workflows that push commits need `permissions: contents: write` and `fetch-depth: 0`
- `workflow_run` events only trigger workflows present on the default branch
- Use `origin/branch` refs (not local) for cross-branch git log comparisons in CI
- Pattern for finding existing PRs: `gh pr list --base X --head Y --state open --json number --jq '.[0].number'`
- VERSION file is the single source of truth for the version string in CI workflows
- Use `actions/upload-artifact@v4` for artifact uploads, `actions/download-artifact@v4` to retrieve them
